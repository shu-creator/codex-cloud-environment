1. 全体像 ─ 抽出品質を担保する処理フロー

fnl-builder は、PDF/CSV のメッセージリスト（ML）から「特記事項」を抽出し、構造化されたゲスト別データとして最終名簿（FNL）に統合するパイプラインである。 品質担保は以下の3段階で行われる。

  1. ルールベース候補抽出（広い網）
    ファイル: src/fnl_builder/parse/messagelist_rules.py
    正規表現キーワード群で「特記事項になりやすい行」を候補として拾い上げる。この段階では取りこぼしを減らすことを優先しており、禁則に該当する行も混在し得る。
  2. LLM による構造化抽出（精密な判定）
    ファイル: src/fnl_builder/llm/prompts/base_system.txt, base_extract.md
    taxonomy に沿って各候補を category/phase/severity 等に分類し、JSON schema に準拠した構造化データを返す。「推測禁止」「evidence完全一致」「禁則優先」ルールにより、虚偽や不要な項目が生成されるリスクを抑制する。
  3. 統合・フィルタ・リライト（最終仕上げ）
    ファイル: src/fnl_builder/pipeline.py, src/fnl_builder/integrate/remark_rewrite.py
    banned パターンによる早期除外、社内メモ・署名の除去、LLM出力とルールベース出力の統合（医療系の信頼度閾値に基づく判定を含む）を行い、最終的にゲスト別の特記事項として名簿に反映する。

パイプライン全体の実行は src/fnl_builder/pipeline.py の run() 関数が起点であり、parse_stage → integrate_stage（内部で LLM 抽出を実行）→ render_stage の順で処理が進む。

2. LLM System Prompt の要約

ファイル: src/fnl_builder/llm/prompts/base_system.txt（v1.0.4, 2026-02-24）
LLM に「ML特記事項抽出エンジン」としての役割を与え、抽出品質を縛る14項目のルールセットを定義している。

2-1. 最重要ルール（全14項目の要約）

No.ルール名内容
1推測禁止入力に書かれていない事実を作らない。常識・経験による補完を禁止。不確実なら explicitness=unclear または severity=caution で表現する。
2JSON出力のみ返信は {"items":[...]} のみ。Markdown、説明文、前置き・後書きは禁止。
3スキーマ厳守taxonomy の id のみ使用（ラベル文字列は使わない）。ML items schema に準拠。
4Evidence完全一致evidence.quote は入力本文からの完全一致コピー。表記（空白、記号、改行）を変えない。正確にコピーできない場合はその item を作らない。
5caution条件severity=caution なら caution_reason 必須。それ以外は空文字。完全一致に自信がない場合のみ evidence_match=false を許容。
6PII最小化旅券番号、電話番号、メール等の個人識別子を who/details/summary/tags に書かない。evidence.quote は根拠として必要だが、個人識別子を含まない短い抜粋を優先。
7抽出判定の順番「手配会社は誰に何をすればよいか」が明確でなければ抽出しない。RQ/HK判定：OPとルーミングは RQ のみ採用、HK/案内/質問は除外。装備サイズは HK でも採用。部屋の位置希望は RQ として accommodation_room で採用。
8禁則金銭/請求/保険/旅券/連絡先変更/社内進行/座席希望/旅行手続書類送付/各地発着申込/時間帯指定なし/未確定RQ/マイページ通知/WT関連/営業メモ等を問答無用で除外。
9航空会社関連機内食RQ/搭乗支援/機内持込配慮/到着影響のみ採用。窓側/通路側/座席指定は除外。
10正規化SSR/WCHR等のコード表記禁止（平文で記述）。新婚旅行/HM/honeymoon は「ハネムーン」に統一。他の祝事は原文表記を使用。
11推測抑制(同室)同室相手が本文で明示された場合のみ抽出。TWN/DBL/TRP が複数並んでも同室ペアを推測しない。
12ランドオンリー正規化L/O, LAND ONLY, ランドオンリー は「ランドオンリー（航空券自己手配・現地合流）」に正規化。合流/離団の場所・日時を summary と handoff_text に含める。
13イレギュラー伝達事項定型外でも「現地オペレーターが行動すべき具体的指示」は採用（VIP贈呈物、特別手配、アップグレード依頼等）。判断基準：手配会社がLANDオペレーターに伝達すべきか。
14振替予約・コース混在時コースNO の主番号（数字部分）で同一性判断。別コースセクションは medical 系のみ採用、OP/宿泊要望/行程関連は除外。

2-2. 禁則の詳細範囲（System Prompt 側）

System Prompt のルール8で定義される禁則は以下の通り。パイプライン側の banned フィルタよりも範囲が広い。

  • 金銭/請求/入金/領収書/クレカ/残金案内
  • 旅行保険
  • 旅券（パスポート番号・発行/有効期限・取得予定等）
  • 連絡先変更（住所/電話/メール/送付先変更）
  • JR等の社内/交通案内、社内進行や一般注意
  • 航空機の座席希望/座席クラス希望（健康理由があっても原則除外）
  • 旅行手続書類の送付希望
  • 各地発着申込
  • 時間帯指定なし（「指定なし」「時間帯指定なし」等）
  • 未確定・予定段階のリクエスト（ただし宿泊要望の「確約せず」「リクエスト中」「希望」は RQ として採用）
  • マイページ単独通知
  • WT（ウェイティング）関連
  • 営業/社内担当者向けの進行メモ・連絡済み報告（「営業へ回送済み」「連絡メモ送信せず」「レラ済み」「イラ済み」「アナ済み」「HC登録あり」「お伺い書待ち」等。ただし同一文中に医療/手配に必要な事実が含まれる場合は営業メモ部分のみ除外し事実部分を残す）
  • 対象者不明のアイテム（who_id を推測で埋めない）

2-3. 強制抽出ルール

以下のキーワード・パターンに該当する記述は、最優先で必ず items 化するよう定められている。

  • アレルギー：「アレルギーなし」等の否定を除き、必ず items 化。category は原則 dietary、phase は meal_time。
  • クレーム/苦情：「クレームなし」等の否定を除き、vip_sensitive として items 化。事実ベースの短い summary/handoff_text を付与。
  • FNL指示文言（「FNL時CHK」「FNLチームへ」等）：対外アクション（現地ガイド/ランドオペレーター/ホテルへの連絡・共有・手配等）が本文に明示される場合のみ items 化。FNL内部確認のみ（「確認しておく」等）では items 化しない。禁則に該当する場合は FNL指示文言があっても除外される（禁則 > 別コーススコープ > FNL指示文言の優先順位）。

2-4. 病気/医療関連の詳細要約ルール

病名・症状に加え、医療機器/治療器具/薬品/量/本数/サイズ/持込方法を handoff_text に具体的に記載する。社内進行メモ（お伺い書待ち/HC登録済み等）は含めない。医療事実・数値・薬品名・機器スペックは省略禁止。240文字制限を超える場合は同一人物について items を複数に分割する。

3. LLM Extract Prompt の要約

ファイル: src/fnl_builder/llm/prompts/base_extract.md（v1.1.0）
System Prompt のルールを前提とし、実際の抽出タスクを指示する User Prompt テンプレートである。

3-1. 入力構造

テンプレートには以下の2つのプレースホルダが存在し、実行時に動的に展開される。

  • {{TAXONOMY_YAML}} ─ taxonomy.yaml の全文がそのまま挿入される（category/phase の正本）
  • {{PAGES_TEXT}} ─ PDF/CSV から抽出したページ別テキスト（1始まりのページ番号付き）

3-2. 出力 schema 要求

各 item に以下のキーを必須とする。

キー型説明
categorystringtaxonomy の categories.id
phasestringtaxonomy の phases.id
summarystring人が即判断できる短文（行動可能な粒度）
explicitnessenumexplicit / implicit / unclear
urgencyenumhigh / medium / low / unknown
confidencefloat0.0〜1.0
severityenumerror / warning / caution
caution_reasonstringseverity=caution なら理由、それ以外は空文字 ""
evidence_matchboolean原則 true
evidenceobject{ page: 整数(1始まり), quote: 完全一致抜粋 }
handoff_textstring運用向けの1文（SSR/WCHRコード禁止）
who_idstring対象者の問い合わせ番号-枝番。特定不可なら空文字 ""

3-3. 出力制約

  • JSON object のみを返す（説明文/Markdown/コードブロック禁止）
  • 全ページを最後まで確認する（後半ページの見落とし禁止）
  • 該当0件なら {"items": []} を返す。無理に作らない
  • handoff_text は運用向けの1文であり、SSR/WCHR等のコード表記は使用しない

4-3. 選び方の指針

  • category は原則として1つに正規化する。迷う場合は other を選び、summary に情報を残す。
  • phase は「実際に対応が必要になる場面」を優先して選ぶ。不明なら unknown を使用。
  • 判断が曖昧でも、必ず evidence（page, quote）に紐づける。根拠なしの推測は禁止。

4-4. スケール定義

explicitness（explicit / implicit / unclear）と urgency（high / medium / low / unknown）はそれぞれ taxonomy 内で定義されており、severity（error / warning / caution）は System Prompt 側で運用ルールが定められている。

5. コース別サプリメントの仕組み

ファイル: src/fnl_builder/llm/prompt_loader.py
コース補足: src/fnl_builder/llm/prompts/courses/417.md, _default.md

5-1. コース番号の抽出ロジック

コースコード（例: E417, EH417, ET470）からアルファベットプレフィックスを除去し、先頭の数字列を「主番号」として抽出する。 正規表現 ^[A-Za-z]{1,3}(\d+) を使用。

  • E417 → 417
  • EH417 → 417
  • ET470 → 470
  • E417Z → 417

5-2. マージ手順

  1. 入力のコースコード群から重複なしの数字列セットを作る
  2. 各数字に対して courses/{number}.md を検索する
  3. 見つかったファイルを数字の昇順に並べ、行単位で重複排除しながらマージする
  4. 1つも見つからなかった場合は courses/_default.md にフォールバックする

5-3. 現存するコースファイル

ファイル内容
417.md シェンゲン協定圏のビザ要件を重点抽出。複数都市移動のため離団・合流の記載に注意。現地ガイドの言語対応を抽出。
_default.md 「コース固有の追加指示はありません。共通ルールに従ってください。」

5-4. プロンプト組み立て

load_prompts() 関数が system prompt（base_system.txt）、extract prompt（base_extract.md）、course supplement をまとめて PromptConfig として返す。course supplement は extract prompt の末尾に追加される形で LLM に渡される。

ファイル: src/fnl_builder/parse/messagelist_rules.py
ML テキストの各行に対して正規表現キーワード群でマッチを行い、「特記事項になりやすい行」を候補として拾い上げる。

6-2. 候補抽出の流れ

  1. 各行に対して _REMARKS_KEYWORDS（30パターン）のいずれかがマッチするかを判定（_has_remark_keyword()）
  2. マッチした行に banned フィルタ（後述）を適用し、該当すれば除外
  3. 残った行について PDF 日本語スペーシング修復、後方メモラベル除去、ダミープレフィックス除去、日付オペレータプレフィックス除去を順に適用
  4. クリーニング後の文字列が3文字を超えていれば候補として返却

6-3. 目的と位置づけ

この段階は「取りこぼしを減らす」ための広い網である。誤検出（偽陽性）は後段の LLM 構造化抽出および統合フェーズで除外されるため、精度よりも再現率を優先する設計になっている。

7. banned フィルタの要約

ファイル: src/fnl_builder/pipeline.py（_BANNED_PATTERNS, _remarks_has_banned()）

7-1. 現状の banned パターン（パイプライン側）

パターン除外対象
請求|入金|領収書|残金|クレジット|クレカ|支払|料金金銭・請求関連
旅行保険|保険旅行保険関連
社内進行|社内手配|社内社内進行メモ関連

7-2. 適用タイミング

_remarks_has_banned() は parse_message_list() 呼び出し時にコールバックとして渡され、ルールベース候補抽出（_extract_message_list_remark()）の中でキーワードマッチ後に呼ばれる。マッチした行は候補から即座に除外される。 また、統合フェーズ（integrate_stage）でも remarks_has_banned としてゲスト統合処理に渡されている。

7-3. System Prompt 禁則との差分

パイプライン側の banned は「最低限の早期除外」に留めており、System Prompt 側の禁則（ルール8）はより広い範囲をカバーしている。 以下は System Prompt にのみ存在する禁則であり、パイプライン側では除外されない。

  • 旅券（パスポート番号・発行/有効期限・取得予定等）
  • 連絡先変更（住所/電話/メール/送付先変更）
  • 航空機の座席希望/座席クラス希望
  • 旅行手続書類の送付希望
  • 各地発着申込
  • 時間帯指定なし
  • 未確定・予定段階のリクエスト（宿泊要望の RQ 例外あり）
  • マイページ単独通知
  • WT（ウェイティング）関連
  • 営業/社内担当者向けの進行メモ（一部は remark_rewrite.py の社内メモ除去で対応）
パイプライン側 banned を System Prompt 禁則に合わせて拡張するか、現状のまま LLM 側に判断を委ねるかは運用方針による。 パイプライン側を拡張すると LLM への入力が減り推論コストが下がるが、禁則管理が二重になるリスクがある。

8. 現場確認ポイント

以下は、コード上の定義だけでは判断が確定しない項目であり、現場担当者との合意形成で運用を安定させるべき論点である。

8-1. 禁則の範囲をどこに寄せるか

  • 現状: パイプライン側の banned は金銭/保険/社内の3パターンのみ。それ以外の禁則は LLM System Prompt に委ねている。
  • 論点: LLM のみに禁則判定を任せると、LLM が禁則を見落とした場合にフォールバックがない。パイプライン側にも旅券/連絡先変更/座席希望等を追加するか。
  • トレードオフ: パイプライン側を拡張すれば安全性が上がるが、禁則定義の二重管理が必要になる。一元管理するなら System Prompt 側を正本とし、パイプライン側はそこからの自動生成にする方法がある。

8-2. RQ/HK の採用基準

  • 現状のルール: OP とルーミング（宿泊要望）は RQ のみ採用。HK/案内/質問は除外。装備サイズは HK でも採用。部屋の位置希望は RQ として accommodation_room で採用。
  • 論点: 「OP申込/HK/予約済」の除外基準が実運用と合っているか。OP の HK（予約確定済み）でも現地に伝達すべきケース（例: 特殊な OP で現地ガイドが把握すべきもの）がないか。

8-3. 別コース混在時の medical 限定採用

  • 現状のルール: 別コースセクションは medical 系（病気/アレルギー/介助/車椅子等）のみ採用し、OP/宿泊要望/行程関連は除外する。
  • 論点: 別コースの VIP/センシティブ情報（クレーム履歴等）も引き継ぐべきか。medical の範囲を medical_health / dietary / mobility_accessibility の3カテゴリに限定するか、baggage_equipment（医療機器）も含めるか。

8-4. evidence.quote 完全一致の厳密度

  • 現状のルール: evidence.quote は入力本文からの完全一致コピーであり、正確にコピーできない場合はその item を作らない。
  • 論点: 完全一致を厳守すると、PDF のスペーシング崩れや OCR 由来の文字化けで正当な項目が採用されないリスクがある。パイプライン側では _repair_pdf_jp_spacing() で CJK 文字間の不要スペースを修復しているが、LLM に渡すテキストにも同じ修復が適用されているか確認が必要。
  • トレードオフ: 完全一致を最優先にすれば採用件数は減るが、虚偽 evidence の混入事故は減る。

8-5. 社内メモ除去の範囲

  • 現状: src/fnl_builder/integrate/remark_rewrite.py で営業メモフレーズ（「営業へ回送済み」「イラ済み」「アナ済み」「レラ済み」「HC登録あり」「お伺い書待ち」等）を正規表現で除去。署名（部署名+担当者名+日時）も4パターンの正規表現で末尾から除去。
  • 論点: 除去パターンの網羅性。新たな社内メモ表現が増えた場合の追加手順を定めておくと運用が安定する。

8-6. 医療系の信頼度閾値

  • 現状: remark_rewrite.py で _MEDICAL_CONFIDENCE_THRESHOLD = 0.80 が定義されている。LLM が返す confidence がこの閾値以上で、かつ evidence_quote が空でない場合に「信頼できる医療アイテム」と判定し、ルールベース候補との統合方式を変える。
  • 論点: 0.80 という閾値が実運用で適切か。閾値を下げると採用件数が増えるがノイズも増える。閾値を上げると確実だが見落としリスクが上がる。

8-7. FNL指示文言の対外アクション判定

  • 現状のルール: 「FNL時CHK」「FNLチームへ」等の文言は、対外アクション（現地ガイド/ランドオペレーター/ホテルへの連絡・共有・手配等）が本文に明示される場合のみ items 化する。
  • 論点: 「対外アクションの明示」の判定基準が LLM の解釈に依存する。実運用で「FNL指示文言があるのに抽出されない」ケースが発生した場合、System Prompt のルールを調整するか、ルールベース側で FNL指示文言を検知して強制候補化するかを決める必要がある。
